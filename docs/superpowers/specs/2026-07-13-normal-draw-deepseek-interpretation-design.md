# 普通解牌 DeepSeek 综合解读设计

日期：2026-07-13  
状态：设计已逐段确认，待规格复核后进入实施计划

## 1. 目标

在不改变现有普通解牌“选择牌阵 → 输入问题 → 逐张抽牌 → 展示牌面与基础牌意”主流程的前提下，在结果页新增用户主动触发的 DeepSeek 综合解读。

成功标准：

- 12 种普通牌阵均能安全生成综合解读。
- 用户只在点击“生成综合解读”后才产生 DeepSeek 请求与费用。
- 解读是一篇连续、完整的中文文章，串联整个牌阵，而不是逐张复述。
- 语气温和清晰，保留塔罗氛围，不宿命、不恐吓、不做医疗、法律或财务保证。
- AI 失败不影响牌面、牌位与基础牌意的查看。
- 现有手势解牌请求保持兼容。

## 2. 范围

### 包含

1. 扩展 `POST /api/readings/interpret`，使其同时支持手势解牌与普通解牌。
2. 新增普通解牌专用请求契约与严格服务端校验。
3. 按牌阵张数动态调整目标文章长度和输出 token 上限。
4. 在普通解牌结果页增加生成、加载、成功、失败和重试状态。
5. 增加外部 AI 数据处理提示。
6. 单元测试、视觉契约测试、构建验证、真实浏览器流程和一次使用虚构问题的 DeepSeek 冒烟测试。

### 不包含

- 购买课程资料的导入、复制、改写、向 DeepSeek 发送或在线上产品中检索。
- 向量数据库、RAG、微调或外部知识库。
- 解读历史、登录同步、分享、收藏或支付。
- 调整普通解牌现有 12 种牌阵的牌位定义。
- 自动重试 DeepSeek 请求。
- 更换模型供应商或新增第三方依赖。

## 3. 资料与版权边界

用户提供的 PDF 与 Word 资料被确认为购买或获得的个人学习资料，不具备商业产品使用授权。本功能因此必须遵守以下边界：

- 不将资料原文、图片、牌阵图或案例复制到仓库。
- 不从资料中生成运行时知识库、检索片段、训练数据或产品文案。
- 不将资料的原文、摘录或案例发送给 DeepSeek 或其他外部模型。
- 本功能只使用项目现有 `lib/tarot-cards.ts` 中的 78 张牌数据和 `lib/normal-draw/spreads.ts` 中的牌阵数据。
- 若未来获得明确商业授权，必须开启新的设计与数据审核阶段，不在本实施中预留隐式导入入口。

参考：[《中华人民共和国著作权法》（中国人大网）](https://www.npc.gov.cn/c2/c30834/202011/t20201119_308796.html)。本节是产品风险控制边界，不替代专业法律意见。

## 4. 统一接口契约

路由保持：

```text
POST /api/readings/interpret
```

### 4.1 现有手势解牌

现有不带 `readingType` 的三张/五张请求继续视为手势解牌，避免破坏已上线的客户端代码。实施时可让新客户端显式发送 `readingType: "gesture"`，但服务端仍必须接受旧格式。

### 4.2 普通解牌请求

```ts
type NormalInterpretationRequest = {
  readingType: "normal";
  question: string;
  spreadId: NormalSpreadId;
  results: Array<{
    cardId: number;
    orientation: "upright" | "reversed";
    slotId: string;
    slotLabel: string;
  }>;
};
```

普通解牌问题在去除首尾空白后必须为 8–240 个字符，与前端现有校验保持一致。

### 4.3 响应

成功响应保持现有结构：

```ts
type InterpretationResponse = {
  interpretation: string;
};
```

失败响应为：

```ts
type InterpretationErrorResponse = {
  error: string;
};
```

面向用户的错误文案不暴露 API Key、环境变量、供应商响应原文或服务器细节。

## 5. 服务端校验与归一化

路由不直接信任浏览器提交的牌阵与牌面数据。验证逻辑抽离为可单元测试的纯函数，返回统一的内部解读输入。

普通解牌校验顺序：

1. `readingType` 必须为 `normal`。
2. `spreadId` 必须能在 `NORMAL_SPREADS` 中找到。
3. `results.length` 必须等于牌阵 `cardCount`。
4. 每个结果必须与相同索引的牌位 `id` 和 `label` 完全匹配，防止牌位伪造与顺序错乱。
5. `cardId` 必须在共享 78 张牌库中存在。
6. 同一牌阵中 `cardId` 和 `slotId` 均不得重复。
7. `orientation` 只允许 `upright` 或 `reversed`。

验证成功后，服务端从本地权威数据源重新解析牌名、关键词、正/逆位基础牌意、牌阵名称、用途和牌位说明；浏览器不得直接提交这些可派生字段作为权威内容。

## 6. DeepSeek 提示词与长度

现有安全系统提示继续作为全流程底线：

- 克制、清晰、有同理心。
- 将整个牌阵视为相互关联的答案。
- 不使用宿命论、恐吓、医疗诊断、法律结论或财务保证。
- 把建议写成可执行的观察与行动。
- 输出必须是合法 JSON，且只包含 `interpretation` 字段。

普通解牌的用户提示词还必须：

- 说明牌阵名称、适用方向和完整牌位顺序。
- 为每个牌位提供本地牌库解析的牌名、正/逆位、基础牌意与关键词。
- 生成一篇连续文章，不使用小标题、列表或 Markdown。
- 不按“第一张、第二张”机械复述，而是串联牌位的变化、张力、资源与方向。
- 在文章末段自然融入 2–3 个可执行建议，不将它们格式化为列表。

目标长度按牌数调整：

| 牌数 | 目标中文字符 | `max_tokens` 上限 |
| --- | ---: | ---: |
| 1–3 | 300–450 | 900 |
| 4–6 | 450–650 | 1200 |
| 7–9 | 650–900 | 1600 |
| 10–12 | 900–1200 | 2200 |

字符范围是提示词目标，不是对模型回复的强制截断条件。服务端对最终 `interpretation` 设置 4000 个字符的防御性硬上限；超过上限时视为供应商响应异常，不在客户端截断后冒充完整解读。

DeepSeek 请求继续使用：

- 模型：`DEEPSEEK_MODEL`，默认 `deepseek-v4-flash`。
- 思考模式：关闭，保持交互速度与可预期成本。
- `response_format: { type: "json_object" }`。
- 不流式输出。
- 超时：28 秒。
- 不使用购课资料、案例或未授权摘录。

## 7. 前端状态与交互

解读请求生命周期由 `NormalDrawExperience` 拥有，`ReadingReveal` 只负责展示状态与触发回调，避免结果视图自己管理路由与业务生命周期。

状态为：

```ts
type InterpretationStatus = "idle" | "loading" | "success" | "error";
```

### `idle`

- 显示完整牌面、牌位和基础牌意。
- 显示主按钮“生成综合解读”。
- 按钮下方显示：“点击后，本次问题与牌面信息将发送至 AI 服务生成解读。”
- 未点击时不发出任何解读请求。

### `loading`

- 按钮显示清晰的生成中状态并被禁用，防止重复扣费。
- 解读容器使用 `aria-live="polite"` 告知辅助技术正在生成。
- 已有牌面和基础牌意保持可见。

### `success`

- 在结果页展示一篇连续的综合解读。
- 同一套牌不显示“再生成一次”，避免无意义重复扣费和多个互相矛盾的答案。
- 如需新解读，用户必须重新抽牌。

### `error`

- 显示温和的可公开错误文案，不暴露 DeepSeek 密钥、额度或服务器细节。
- 显示“重新生成”按钮，只有用户点击后才再次请求。
- 本地牌面与基础牌意保持完整。

### 生命周期安全

- 生成中重新抽牌、更换牌阵、返回问题或卸载页面时，中止当前 `AbortController`。
- 每次请求使用递增请求 ID；过期响应即使晚到也不得更新新牌阵。
- 重新抽牌和更换牌阵均清空解读状态、错误和文本。
- 请求指纹由问题、牌阵与所有牌位结果组成，不允许同一套牌在 `loading` 或 `success` 时重复发送。

## 8. 错误处理

服务端保持现有错误分类：

- 请求不是合法 JSON：`400`。
- 牌阵、牌位、卡牌或问题数据不合法：`400`。
- 服务器未配置 Key：`503`，对外文案不透出配置细节。
- DeepSeek `401/402`：服务器记录可诊断原因，客户端只收到通用错误。
- DeepSeek `429`：返回可重试的繁忙提示。
- 28 秒超时：返回可重试的超时提示。
- DeepSeek 空内容、非法 JSON 或缺少 `interpretation`：按供应商失败处理，不自动重试。
- 解读超过 4000 个字符：拒绝渲染并返回通用错误，防止异常长回复影响客户端。

所有失败都必须保留已抽牌面和基础牌意，AI 不是查看结果的必要依赖。

## 9. 测试与验收

### 纯逻辑测试

1. 12 种普通牌阵的合法请求都能通过归一化。
2. 未知牌阵、缺牌、多牌、错误顺序、错误牌位 ID/名称、重复卡牌、重复牌位、未知卡牌、非法正逆位均被拒绝。
3. 普通问题的 8–240 字符规则与前端一致。
4. 旧手势请求仍能通过验证。
5. 四个牌数区间生成正确的目标长度与 `max_tokens`。
6. 普通提示词包含全部牌位、牌名、正逆位和牌位说明，且明确要求连续文章、非 Markdown、非逐张复述。
7. 空内容、非法 JSON、空 `interpretation` 和异常过长文本被拒绝。

### 前端契约测试

1. 结果页保留基础牌意。
2. `idle` 状态包含“生成综合解读”和外部 AI 数据处理提示。
3. `loading` 状态禁用重复点击。
4. `success` 状态渲染一篇连续解读。
5. `error` 状态提供“重新生成”，且错误文案不包含 DeepSeek、Key 或配置细节。
6. 重新抽牌、更换牌阵和返回问题会清理解读生命周期。
7. 手势解牌的现有视觉契约不受影响。

### 完整验证

- `npm test`
- `npx tsc --noEmit`
- `npm run lint`
- `npm run build`
- 桌面浏览器走通：选牌阵 → 问题 → 抽完 → 点击生成 → 显示综合解读。
- 390×844 移动端验证按钮、隐私提示、加载状态和长文滚动，无页面横向溢出。
- 使用一组虚构问题和项目现有牌面数据执行一次真实 DeepSeek 冒烟测试，不读取、引用或发送用户的购课资料。
