# 普通解牌：选阵后提问与逐张抽牌设计

日期：2026-07-12  
状态：设计已选定“逐张亲手抽牌”，规格已自审，待用户复核后实施

## 1. 目标

把首页第一个入口“普通解牌”落到 `/draw`，形成一条与每日一牌、手势抽牌明确隔离的连续流程：

`选择牌阵 → 确认牌阵与牌位 → 输入问题 → 逐张抽牌 → 完成牌阵`

本轮重点不是制作普通表单，而是把“已选牌阵、适用范围、牌位结构、问题输入和进入抽牌”组织成同一场黑金银河仪式，同时保证信息清楚、键盘可用、移动端无页面横向滚动，并尊重 `prefers-reduced-motion`。

## 2. 已确认事实与约束

- 当前 `app/draw/page.tsx` 仍是占位页，12 牌阵选择页仅存在于视觉原型 `obsidian-galaxy-motion-v6.html`，尚未进入生产代码。
- 普通解牌固定使用 `/draw`，不引入 `/gesture-draw` 的摄像头、MediaPipe、手势阈值或手势状态机。
- 可复用 `lib/tarot-cards.ts` 的 78 张牌数据、正逆位牌意和现有卡背视觉，但不得修改每日一牌或手势抽牌现有的随机逻辑。
- 现有 `/api/readings/interpret` 只接受手势流程的三张/五张牌阵。本轮不修改其请求类型、校验、DeepSeek 地址、模型参数或错误处理，也不把 12 牌阵错误发送给该接口。
- 视觉延续已确认的“黑曜石银河＋象牙白/银白/暖金粒子＋精密天文仪器”方向；蓝色只作为低亮辅助轨，不回到蓝色宇宙主题。
- 牌是抽牌阶段的视觉主角。星盘、符印和粒子只负责解释牌位与建立空间深度，不能遮住牌或文字。

## 3. 本轮范围

### 包含

1. 将 12 牌阵图鉴落入生产 `/draw`。
2. 点击牌阵后进入同一路由的“牌阵确认与问题输入”阶段。
3. 清楚展示已选牌阵、适用范围、张数和所有牌位含义。
4. 校验问题并进入普通点击抽牌阶段。
5. 从独立洗好的 78 张牌堆中逐张选择，牌飞入当前牌位；不得重复抽到同一张牌。
6. 最后一张入位后统一揭示牌面，并显示每张牌对应的牌位、正逆位与基础牌意。
7. 为后续综合解读保留稳定的 `NormalReadingResult[]` 数据契约，但本轮不调用或修改 DeepSeek 接口。

### 不包含

- AI 综合解读接口扩展、牌灵聊天、历史记录、登录同步或分享。
- 摄像头、手势识别、Three.js 或手势页面组件复用。
- 每日一牌的本地锁定逻辑。
- 分类筛选、搜索、收藏牌阵或新增超过 12 个牌阵。
- 新增第三方依赖。

## 4. 牌阵定义与校准

每个牌阵必须有稳定 `id`、名称、张数、适用说明、牌位名称、短释义和百分比坐标。牌位坐标既驱动确认页的星图，也驱动抽牌后的落位舞台。

1. **单牌指引（1）**：核心指引。适用于快速确认一个明确问题的核心提示。
2. **三牌时光（3）**：过去、现在、未来。适用于梳理一件事的发展脉络。
3. **四季牌阵（4）**：萌芽、盛放、收获、沉潜。适用于观察一个项目、关系或个人周期的四个阶段；不是机械对应自然季节预测。
4. **关系牌阵（5）**：你的状态、对方状态、关系核心、主要阻力、发展方向。适用于理解双方互动与关系走向。
5. **决策牌阵（5）**：选择 A 的机会、选择 A 的代价、选择 B 的机会、选择 B 的代价、判断原则。适用于比较两个明确选项，不替用户做决定。
6. **十字牌阵（5）**：问题核心、主要阻碍、可用资源、行动建议、可能结果。适用于拆解单一但有阻力的问题。
7. **马蹄铁阵（7）**：过去、现在、隐性影响、主要阻碍、外部环境、建议、可能结果。适用于追踪多因素事件的演变。
8. **月相牌阵（8）**：意图、萌芽、行动、显化、看见、释放、整合、更新。适用于情绪周期、习惯改变与阶段性成长。
9. **灵魂探索（9）**：外在人格、核心渴望、阴影、恐惧、天赋、重复模式、当前课题、整合方向、下一步。适用于深入理解内在模式，不用于心理诊断。
10. **凯尔特十字（10）**：现状、挑战、根基、过去、可能性、近期发展、自我、环境、希望与恐惧、可能结果。适用于长期且相互牵连的复杂问题。
11. **年度十二宫（12）**：自我、资源、沟通、家庭、创造、日常与健康、关系、共享与转化、信念与远行、事业、社群、潜意识。适用于观察未来一年十二个生活领域的主题；它不是“未来十二个月一月一张”，避免名称与牌位逻辑冲突。
12. **事业发展（6）**：职业现状、核心优势、盲点、机会、阻力、下一步行动。适用于职业选择、成长与行动规划。

## 5. 页面与交互

### 5.1 牌阵图鉴阶段

- `/draw` 默认展示 12 牌阵图鉴。
- 每个可点击符印只显示编号/张数、牌阵名称和一行“适用于……”，不出现问题输入框。
- hover、键盘 `focus-visible` 和选中态统一驱动整组轨道旋转、节点错峰呼吸、卡片轻抬与暖金辉光；选中后状态锁定。
- 点击后先锁定选中态 240ms，再用 Next Router `push` 写入 `?spread=<id>`，使刷新、复制地址和浏览器返回可恢复阶段；reduced-motion 下跳过等待并立即进入下一阶段。
- 无效或缺失 `spread` 参数回到图鉴，并提供可理解的状态提示，不渲染空白页。

### 5.2 牌阵确认与问题输入阶段

桌面采用“星图主舞台＋问题铭刻台”的非对称布局，移动端改为单列顺序：

1. 顶部保留粒子品牌与“返回牌阵图鉴”。
2. 主舞台放大已选牌阵符印，显示名称、张数和适用范围。
3. 牌位结构以真实坐标展示；每个节点同时有编号和短标签。旁边或下方提供完整牌位列表，避免只靠图形理解。
4. 问题输入位于切角暖金框和细轨道组成的“铭刻台”中，不做普通白底或 dashboard 表单。
5. 问题长度为去除首尾空白后 8–240 个字符；纯数字、纯符号或明显重复字符给出具体错误。错误通过 `aria-describedby` 关联，不能只用颜色。
6. 提交按钮文案为“开始抽牌”，未满足校验时不静默失效；提交后把焦点移动到抽牌阶段标题。
7. “更换牌阵”返回图鉴；问题尚未提交时无需保存。进入抽牌后返回确认页时保留本次问题和牌阵，但清空已抽结果，避免半套牌阵产生歧义。

### 5.3 逐张抽牌阶段

- 进入阶段时只洗牌一次，得到 78 张不重复的随机牌序；现有每日一牌和手势抽牌随机函数不作任何修改。
- 每次展示一组可点击牌背候选：宽度大于 680px 时显示 18 张，680px 及以下显示 12 张。候选牌映射到剩余牌堆中的真实唯一卡牌，点击哪一个就选择其对应卡牌，并从剩余牌堆删除；落位完成后从剩余牌堆补齐候选窗口。
- 界面同时显示“牌堆剩余 N 张”，不声称 78 张牌背会同时铺开。牌背最小点击区域为 44×44px，候选区在容器内重排，页面本身不能横向滚动。
- 页面明确显示“第 N / 总张数张”和当前牌位名称。选中牌从候选区飞向当前牌位，完成落位后才允许选择下一张，防止快速连击重复入位。
- 每张牌在入位时独立生成 50/50 正逆位，并写入结果；不修改项目其他流程的正逆位逻辑。
- 抽牌前所有牌面保持隐藏。最后一张稳定落位后进入统一揭示，降低星盘亮度，让牌面成为主角。
- reduced-motion 下取消飞行、旋转和粒子拖尾，直接完成状态切换，但仍保留选中、进度和焦点反馈。

### 5.4 完成牌阵阶段

- 统一揭示所有牌，展示牌位、牌名、正/逆位及对应基础牌意。
- 页面不发送 DeepSeek 请求，也不伪装成“AI 综合解读已完成”。清楚区分“牌面与基础牌意”与后续“综合解读”。
- 提供“重新抽牌”和“更换牌阵”。重新抽牌会生成新的牌序并清空本次结果；更换牌阵返回图鉴。

建议新增独立目录，避免继续膨胀 `app/draw/page.tsx`：

```text

components/draw/

  CardDrawStage.tsx          # 候选牌背、进度与落位
  ReadingReveal.tsx          # 完成牌阵与基础牌意
  NormalDraw.module.css      # 本流程视觉与响应式规则
lib/normal-draw/
  spreads.ts                 # 12 牌阵与牌位定义
  state-machine.ts           # 普通点击流程状态机
  deck.ts                    # 独立洗牌、候选与正逆位纯函数

  types.ts                   # 稳定数据契约
```

核心阶段：

```text
ATLAS → QUESTION → DRAWING → REVEAL
```

核心结果：

```ts
type NormalReadingResult = {
  cardId: number;
  orientation: "upright" | "reversed";
  slotId: string;
  slotLabel: string;
};
```

普通流程只依赖共享牌库，不依赖 `lib/gesture/*`。如果符印几何算法未来确实可以共享，应先抽成不包含业务状态的纯视觉工具；本轮不为“可能复用”提前重构手势代码。

## 7. 可访问性与响应式要求

- 所有牌阵、牌背、返回和提交操作必须是原生 `button` 或 `a`，具备明确可访问名称。
- `focus-visible` 必须与 hover 有等价可见反馈；不能隐藏 outline 而不给替代焦点环。
- 阶段标题使用语义化 heading；进度与错误使用 `aria-live`，但避免每个粒子或动画进入可访问树。
- 390×844 下页面宽度不得超过视口；牌位列表可以纵向增长，不能通过缩小文字解决拥挤。
- 正文和输入文字需达到可读对比度；暖金不用于大段低字号正文。
- `prefers-reduced-motion: reduce` 下停用轨道持续旋转、节点脉冲、银河呼吸、鼠标粒子和牌面飞行动画。

## 8. 错误与恢复

- 无效牌阵参数：返回图鉴并提示“该牌阵不存在，请重新选择”。
- 问题不合格：留在当前阶段，聚焦错误字段并说明原因。
- 抽牌阶段数据不一致：安全回到问题阶段并清空半成品，不尝试补造结果。
- 卡牌数据缺失：完成页显示明确错误和“重新抽牌”，不得把 `undefined` 牌面渲染给用户。
- 页面刷新：通过 `spread` 参数恢复到确认页，不尝试在没有持久化契约时恢复半套抽牌。

## 9. 测试与验收

### 自动化测试

1. 12 牌阵的 `id` 唯一、张数与牌位数量一致、牌位 `id` 唯一、坐标在有效范围。
2. 年度十二宫使用生活领域牌位，不被误写成十二个月。
3. 问题校验覆盖空白、过短、过长、纯数字、纯符号、重复字符与合法中文问题。
4. 洗牌结果包含 78 个唯一牌 ID；逐张选择不重复；正逆位只产生合法值。
5. 状态机拒绝跨阶段动作、锁住落位中的重复点击、最后一张后进入 `REVEAL`、重置后清空结果。
6. 视觉契约覆盖 12 牌阵、键盘焦点、reduced-motion 和移动端无横向页面溢出所需的关键结构/样式。
