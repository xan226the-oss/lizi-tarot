# 牌库 A 方案：固定视口阅读与 78 张原创解读设计

**状态：** 已确认设计；尚未授权实施
**日期：** 2026-07-17
**设计基准：** 桌面 1440 × 900
**视觉方向：** A「紧凑档案台」

## 1. 目标与不可变边界

牌库必须把“78 张牌”、一行筛选与首排卡面放进桌面首屏，并把单张详情改为固定在一个桌面视口内完成阅读的档案界面。详情右侧只在“牌义”和“场景”两个完整状态间原位切换；切换不得移动左侧牌面、标题、关键词或前后牌导航。

此设计只覆盖以下边界：

- `/library`、`/library/[cardId]`；
- 与牌库直接相关的组件、样式、测试，以及只被牌库读取的牌义数据；
- 牌库所需的原创文案与其来源记录。

明确不改：普通抽牌、手势抽牌、聊天、DeepSeek、全局导航行为、现有卡图资源、URL 筛选语义、Git 状态。不得重置、覆盖现有未提交改动，不提交、不推送、不建 PR。

`lib/tarot-cards.ts` 是普通抽牌也会读取的共享基础数据；为避免一次“补全牌义”意外改变普通抽牌，它不是本次文案扩写的落点。牌库应新增独立的只读解读数据模块，供 `/library/[cardId]` 使用。

## 2. 首页：紧凑档案台

### 桌面结构

1. 保留现有 64px 品牌栏和“返回首页”入口。
2. 紧随其后的是单行档案身份区：`档案目录 / 牌库 / 78 张`。不再保留大标题下的世界观说明、空白英雄区或重复说明文。
3. 搜索、阿尔卡纳、花色、清空保持为一行紧凑控制区，保留现有 URL 驱动状态、浏览器历史恢复和无请求的本地筛选。
4. 结果数紧随控制区，首排牌卡在同一首屏可见。1440px 宽的验收基准展示 6 列；较窄但仍为桌面的断点可依次降为 5 / 4 列，不能用横向页面滚动换取列数。
5. 卡片仍展示图、中文名、英文名、类别和关键词；键盘焦点、空结果、失败图回退、动效减弱支持保持不变。

### 移动端

768px 以下允许筛选换行或控件内部横向滚动，网格自然降列。首页的“首屏尽快出现牌”目标不以压缩触控尺寸为代价，按钮和输入框仍至少 44px 高。

## 3. 详情：一个稳定阅读框架，两个互斥完整状态

### 桌面固定视口合同

桌面验收目标固定为 1440 × 900：

- 页面壳、品牌栏、面包屑和详情主体共同占满视口；在该基准下页面不产生纵向滚动条。
- 详情主体为 `22rem + 自适应右栏` 的两栏布局。左栏展示 2:3 牌面，始终是阅读锚点。
- 右栏是一个垂直网格：牌的身份信息、状态切换、可用内容区、前后牌导航。内容区不得设截断高度、`overflow: hidden`、省略号或内嵌滚动条。
- 字体正文最小 16px，行高不低于 1.7；任何“单屏”目标都不得通过缩小正文、裁切或隐藏内容实现。

800 × 600 等矮桌面无法同时承载完整可读文本和“永不滚动”。本次确认的约束只以 1440 × 900 为无滚动验收基准；高度低于约 780px 时，允许恢复自然页面滚动以保证内容完整，不能伪造单屏效果。

### 共同、不切换的区域

右栏顶部始终显示：档案编号与类别、中文名、英文名、关键词。右栏底部始终显示前一张 / 后一张导航，并沿用首尾边界的禁用状态，不循环。

### 状态一：牌义

默认状态是“牌义”。它不是现有“正位 / 逆位”二选一标签页，而是同屏呈现两块完整可读的并列解读：

- 左块：正位标题和完整原创正文；
- 右块：逆位标题和完整原创正文；
- 两块之间只在宽屏下并列；窄屏则按照正位、逆位自然纵向排布；
- 不新增另一个会隐藏文案的“正位 / 逆位”切换层。

这让用户能直接比较正逆位，并把唯一的交互层级保留给“牌义 / 场景”。

### 状态二：场景

点击“场景”后，只替换右栏内容区。它完整显示：

1. 与牌面已有原创设定一致的场景记录；
2. 人物；
3. 地点；
4. 符号。

后面三项在桌面显示为三列档案项；移动端恢复为正常纵向顺序。不得把场景正文折叠进“展开更多”。

### 可访问性与行为

- “牌义 / 场景”使用一个本地、无 URL 副作用的 ARIA tablist；有正确的 `role=tab`、`aria-selected`、`aria-controls`、`role=tabpanel` 与唯一 ID。
- 鼠标点击与左右方向键都可切换；切换后焦点落在被选标签上。
- 切换只改变内容区，不触发页面跳转、路由请求、数据请求或滚动定位。
- 卡图、所有可点控件、导航和当前样式的 `prefers-reduced-motion` 支持继续保留。

## 4. 78 张原创正逆位解读的数据合同

### 新数据边界

新增牌库专用数据模块 `lib/tarot-library-readings.ts`，导出按 `cardId` 索引的 78 条记录。每条记录仅包含：

```ts
type TarotLibraryReading = {
  cardId: number;
  upright: string;
  reversed: string;
};
```

详情路由读取这份数据并传给牌库专用详情面板。它不修改 `TarotCard` 的共享 `meaning_upright` / `meaning_reversed` 字段，也不改变普通抽牌或 DeepSeek 的输入输出。

### 原创与依据规则

每一张的正位与逆位都必须是独立撰写的完整解读，不得复用同一花色的模板句后只替换牌名。文案唯一可依据以下项目：

- 该牌已有的中文名、关键词、正逆位基础含义；
- 该牌已有的 `story`、`sceneSummary`、人物、地点、符号和牌面 alt；
- 已确立的原创“中西融合星际神话”世界观。

不得复制外部牌义、书籍、网站或现成牌组文字；不得给既有场景添入没有资料支持的人物关系、历史事实、地点设定或因果。可以把已有线索转化为面向读者的判断、提醒和行动建议，但不能把推断当成牌面事实。

为使完整内容在 1440 × 900 基准内保持可读，每段建议为 90–140 个汉字，最大不得超过 160 个汉字；正位与逆位均须有具体的状态判断与可执行的阅读提示。场景记录使用现有的完整资料，若需要补写，必须先在来源记录中列明可用线索。

### 来源记录与质量闸门

同时新增只读数据文档 `docs/tarot-library-reading-source-map.md`：每张牌列出卡号、可用关键词、已有场景线索，以及“仅据这些线索扩写”的记录。该文档用于人工审读原创性与依据，不面向运行时读取。

在代码测试外增加数据完整性验证：

- 记录数严格为 78，`cardId` 覆盖 1 到 78 且无重复；
- 每张同时有非空正位与逆位，两段均在内容长度上限内；
- 78 张正位、78 张逆位分别不能出现完全重复文本；
- 每张牌库条目都有场景资料；
- 任一缺失都视为牌库构建错误，而不是以共享短牌义或外部请求静默回退。

这些自动检查只能验证结构与重复，不能证明文学原创性；78 张逐条“关键词 / 既有牌义 / 场景资料 / 新解读”的人工审读是交付门槛。

## 5. 组件与数据流

```text
TarotDetailPage (server)
  ├─ getTarotCardById(cardId)                  → 共享基础名称、关键词、分类
  ├─ getTarotLibraryEntry(cardId)              → 现有原创牌面与场景资料
  ├─ getTarotLibraryReading(cardId)            → 新增的牌库专用完整正逆位解读
  ├─ TarotCardArtwork                          → 左栏固定牌面
  ├─ TarotDetailContent (client, local state)  → “牌义 / 场景”原位切换
  │    ├─ MeaningView                           → 正位 + 逆位同时完整显示
  │    └─ SceneView                             → story + 人物 / 地点 / 符号
  └─ TarotDetailNavigation                     → 右栏底部、不循环的前后导航
```

`LibraryBrowser` 继续独占首页筛选的 URL 与历史状态。详情切换只在新增的 `components/library/TarotDetailContent.tsx` 内管理，不写 URL、不调用 API、不影响其它路由。`app/library/[cardId]/page.tsx` 改为渲染该组件；现有 `TarotMeaningTabs.tsx` 在同一变更中删除，避免保留双层 tab 结构。`TarotStoryPanel.tsx` 保留为场景状态的展示组件。

## 6. 失败处理

- 无效卡号、无基础卡数据或无场景数据继续走现有 404。
- 有合法卡号但缺牌库完整解读属于发布前数据不完整，应在构建/测试阶段失败；运行时不以短文案、通用文案或网络内容替代。
- 图片加载失败继续使用现有本地回退，不影响牌名、牌义或场景阅读。
- 内容超出单屏预算时，先修改相应原创文案使其在准确性和篇幅上达标；禁止通过 CSS 裁切解决。

## 7. 验收与测试

### 自动检查

- 保留并更新牌库筛选、浏览器历史、图片回退、静态路由、404、导航边界与无 DeepSeek 依赖的现有测试。
- 在 `tests/tarot-library-visual-contract.test.mjs` 新增两状态切换的 ARIA、键盘和本地状态测试。
- 新增 `tests/tarot-library-readings.test.mjs`，验证 78 条阅读资料的覆盖、长度、唯一性和卡号完整性。
- 新增样式合同检查：1440px 桌面首页可达 6 列；详情 22rem 左栏；仅一个“牌义 / 场景”tablist；牌义双栏；场景三列；没有用于正文的截断 / 内滚动规则。

### 人工浏览验收

1. 1440 × 900：打开 `/library`，首屏看见“牌库 / 78 张”、完整一行筛选、结果数与 6 张首排牌；无大块英雄空白。
2. 1440 × 900：打开 `/library/1` 及最长文案样本，页面无纵向滚动，牌面、两段正逆位、场景、前后牌导航均无裁切。
3. 切换“牌义 / 场景”后，左侧牌面与右栏身份区位置不变；每个状态内容完整可读。
4. 1366 × 768 与低于约 780px 高度：检查不会裁切；允许自然页面滚动以保护正文。
5. 768px 以下：检查单列、触控尺寸、文字折行、两段牌义与场景资料的顺序完整性。
6. 抽样审读全部 78 张，逐条比对来源记录；不通过原创性或依据检查的文案不能进入运行时数据。

## 8. 实施顺序

1. 先建立阅读数据与来源记录，并完成 78 张逐条审读；
2. 为数据完整性写失败测试；
3. 实现首页紧凑布局；
4. 实现详情两状态组件与固定视口样式；
5. 更新视觉合同和可访问性测试；
6. 完成桌面 / 移动人工验收；
7. 保持所有改动未提交，等待用户决定后续 Git 操作。
