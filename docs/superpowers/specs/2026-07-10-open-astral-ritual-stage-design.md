# 开放式星盘仪式台设计

日期：2026-07-10

## 目标

重做牌阵选择、连续浏览、候选牌反馈和全部选牌后的承托视觉，使体验从“大型玻璃弹窗中的功能选择”转变为“直接发生在星空中的完整占卜仪式”。保留深海蓝、暖金、象牙白和现有卡背资产，不修改抽牌随机性、牌阵数据或手势识别阈值。

## 已确认方向

- 视觉风格：开放式星域仪式，不保留当前中央大灰色面板。
- 完成底座：三张或五张牌共用一整座大型星盘投影祭坛，不为每张牌单独创建小底座。
- 动效原则：反馈清楚，但不使用大面积白光覆盖牌背；运动连续，不让边缘新牌重复播放入场动画。

## 视觉系统

### 色彩

- 黑曜深空 `#050916`：页面基底和卡背暗部。
- 午夜蓝 `#0A1730`：局部雾层和内容承托。
- 仪式暖金 `#D6B45A`：选中描边、星盘主轨和关键状态。
- 象牙白 `#F4EBDD`：标题和主要文字。
- 星雾蓝 `#6D86B5`：次级轨道、体积雾和空间纵深。

### 字体

- 中文标题继续使用现有 `Noto Serif SC`，只在页面标题和牌阵名称中使用。
- 英文与星盘标记继续使用现有 `Cinzel`。
- 功能说明与状态提示使用当前正文无衬线字体，避免所有文字都呈现为装饰性标题。

### 签名元素

共享星盘投影祭坛是本轮唯一的强视觉元素。其他光晕、边框和粒子必须服从它，不能让每个区域都拥有同等强度的装饰。

## 牌阵选择页

### 结构

移除 `SpreadSelection` 中的中央 `recorder` 大面板。页面改为开放式三层结构：

1. 顶部保留品牌、返回首页和摄像头模式操作。
2. 中央显示紧凑标题以及两个牌阵星图符印。
3. 底部显示单行问题摘要和低权重的仪式提示。

```text
顶部：品牌 / 返回首页                         摄像头状态

                         选择牌阵
                 三张星轨          五芒星阵
                 过去 现在 未来    身 心 灵 情感 环境

                 ───── 你的问题 ─────
```

### 牌阵选项

- 三张牌使用横向弧形星轨，三个牌位由细线连接。
- 五张牌使用五芒星轨，五个牌位沿星形顶点分布。
- 选项不再表现为两个大型矩形卡片，而是两个透明的仪式符印区域。
- 默认状态只显示低亮轨道；悬停或键盘聚焦时，牌位依次点亮、轨道缓慢旋转，标题提升至暖金。
- 点击区域保持足够大，视觉上开放但交互上仍是完整按钮。

### 摄像头错误

- 摄像头初始化失败不再占据页面中央。
- 错误改为顶部操作区下方的小型状态条，说明已经切换到点击模式并保留“重试摄像头”操作。
- 错误状态使用低饱和玫瑰灰边框，不使用高亮红色大框。

## 选牌反馈

### 候选牌状态

食指或点击指针当前命中的牌必须同时获得四个反馈：

1. 1.5px 暖金外描边。
2. 牌背外侧的椭圆呼吸光，不覆盖牌背内部图案。
3. 约 6% 的前移和不超过 4% 的放大。
4. 少量星尘沿卡牌边缘向上流动。

指针本身保持为小型金色准星，并位于卡牌前方；它只表示手指位置，不承担选中状态的主要表达。候选牌离开命中范围后，在 180–240ms 内回到普通状态。

### 抓取确认

- 捏合开始后，暖金描边变为稳定常亮，呼吸光收紧到卡牌边缘。
- 确认完成时只播放一次沿边框闭合的金色光流，不使用整张牌闪白。
- 取消抓取时平滑回到候选状态，不突然熄灭。

## 连续牌环

### 根因

当前 `gesture-card-orbit--expanding .gesture-tarot-card` 会对所有新挂载的边缘卡牌重复应用 `gesture-card-summon`。横向连续移动时，每进入一张新牌就重新播放缩放与入场延迟，因此产生“一张张插牌”的感觉。边缘牌的基础透明度又高于 50%，进一步放大了跳入感。

### 设计修正

- 扇面首次进入 `SELECTING` 时，仅让整个牌环容器播放一次展开动画。
- 连续浏览阶段，单张卡牌不再拥有入场动画。
- 左右边缘各使用约 1.5 张牌宽度的透明度渐隐区，新牌从透明连续过渡到正常透明度。
- 牌位 transform 继续由现有逐帧轨道算法驱动，不增加 CSS transform 过渡，以免与 requestAnimationFrame 相互拉扯。
- 速度阻尼保留，但需要为持续指向最左或最右增加平稳的最高速度区，避免到达边缘后继续加速。

## 共享星盘投影祭坛

### 出现时机

- 前几张牌落位时只显示克制的牌位能量核心。
- 最后一张牌确认后，牌环整体变暗并进入 `CONFIRMED`。
- 共享祭坛在确认停顿中由下向上显现，并在 `REVEALING` 时达到最高亮度。
- 结果页保留亮度约 25% 的祭坛，使揭示与解码界面具有连续的空间关系。

### 构成

祭坛完全使用 CSS 层构成，不使用图 2 作为贴图：

- 三层扁平透视椭圆轨道，速度和旋转方向不同。
- 中央接触光与牌阵形状一致：三张为横向弧轨，五张为五芒星轨。
- 一束宽而柔的纵向体积光承托整组牌，不形成舞台射灯感。
- 低密度星尘和星云薄雾只集中在祭坛周围。
- 祭坛位于卡牌后方、普通背景前方，绝不覆盖牌面文字或牌背图案。

### 三张与五张适配

- 三张牌祭坛宽而扁，主轨为水平椭圆，三个接触点沿中轴排列。
- 五张牌祭坛更接近圆形，五个接触点沿五芒星顶点分布。
- 两种模式共用一个 `RitualPlinth` 组件，通过 `spread` 决定轨道比例和接触点布局。

## 组件边界

- `SpreadSelection`：开放式页面结构、牌阵符印和摄像头状态，不管理抽牌过程。
- `CardOrbit`：首次展开标记、连续卡牌布局和候选牌状态。
- `TarotCard3D`：牌背、候选描边、抓取确认和边缘星尘。
- `EnergyPointer`：只显示输入位置和抓取状态，不再替代卡牌选中反馈。
- `RitualPlinth`：共享祭坛视觉；只消费 `phase`、`spread` 和结果数量，不派发业务动作。
- `TarotUniverse`：编排牌环、牌位和祭坛的视觉层级。
- `ResultSpace`：在结果牌阵后方复用低亮度祭坛。

## 数据流与状态

- 不新增业务状态；继续使用 `SELECTING`、`GRABBING`、`PLACING`、`CONFIRMED`、`REVEALING` 和 `RESULT`。
- 牌环首次展开属于视觉状态，可由 `CardOrbit` 在进入 `SELECTING` 时记录，不写入业务 reducer。
- 候选牌 ID 继续来自现有 `OrbitFrame.selectedCardId`。
- `RitualPlinth` 根据 `phase` 决定 `hidden / emerging / active / ambient` 四种视觉强度。

## 无障碍与性能

- 牌阵选项保留原生按钮语义和清晰的 `focus-visible` 状态。
- `prefers-reduced-motion` 下停止轨道旋转、星尘和呼吸动画，但保留静态描边与选中对比。
- 不新增图形或动效依赖；星盘、雾和粒子优先使用伪元素、渐变和 transform/opacity。
- 连续浏览不得因 React 新挂载边缘卡牌而触发单牌动画。

## 测试与验收

1. 牌阵选择页不再出现中央大型灰色面板，三张与五张以开放式星图符印呈现。
2. 鼠标、点击模式和手势模式中，当前候选牌都能通过描边、外光和轻微前移被清楚识别。
3. 持续向左或向右浏览 5 秒，边缘卡牌连续淡入，不出现逐张缩放插入。
4. 三张或五张全部选完后，只出现一个共享祭坛，并与对应牌阵形状匹配。
5. 祭坛不会覆盖牌背或结果文字，结果页只保留低亮环境态。
6. 1280×720、1536×821 和移动端宽度下无页面级意外滚动或核心内容裁切。
7. 减少动态效果模式下仍能辨认选中牌和牌阵选择。
8. 自动化测试、TypeScript、Lint 和生产构建通过。

## 非目标

- 不调整 MediaPipe 手势分类与置信度阈值。
- 不修改抽牌随机性、正逆位和 DeepSeek 解读接口。
- 不引入 Three.js、R3F、Framer Motion 或新的图片生成资产。
- 不重做首页、问题输入页和与本次流程无关的其他页面。
