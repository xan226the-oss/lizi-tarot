# 手势抽牌全屏重做与 DeepSeek 综合解读设计

日期：2026-07-10

## 目标

同时重做三张与五张手势牌阵的洗牌、选牌、落牌、统一揭示和结果体验，并接入 DeepSeek 生成面向用户问题的综合解读。保留现有状态机、手势控制层与视觉层的职责边界，不让 MediaPipe 或视觉组件直接管理业务状态。

本设计同时包含此前审查确认的四项缺陷修复：点击模式牌环动画、移动端摄像头控制、摄像头帧引发的无效整页渲染、每日一签异常存储恢复。

## 实测结论

三张与五张牌阵共用 `TarotUniverse`、`CardOrbit`、`SpreadBoard` 和 `ResultSpace`，因此用户指出的五项问题在两个牌阵中都存在：

1. 最后一张牌会被单独抽离并揭示牌意，造成它像是唯一答案；五张牌阵中同样表现为第五张被突出。
2. 洗牌卡牌偏小，运动范围集中在画面中部。
3. 选牌扇面未充分占据主舞台，结果页又把牌阵压缩到小区域，缺少沉浸感和视觉冲击力。
4. 选中牌通过牵引线和巨大单牌抽离表达，破坏扇面整体性。
5. 已抽牌位使用 `card-slot-mist.png`，呈现为贴图而非实时能量效果。

## 最终流程

### 洗牌

- 保留 `SHUFFLING` 业务状态，但把牌群放大约 1.4 到 1.7 倍。
- 卡牌在主舞台大范围穿梭、交错和聚拢，运动范围覆盖接近整个可用宽度。
- 洗牌结束后牌群展开为宽幅扇面，不突然缩回中央。

### 选牌

- 扇面横跨约 90% 主舞台宽度，卡牌尺寸随视口响应，桌面端明显大于当前版本。
- 保留横向无限浏览与指针选牌算法。
- 当前候选牌只使用金色流光、边缘辉光、轻微前移和呼吸亮度表达。
- 删除能量牵引线。
- `GRABBING` 阶段不再把牌从扇面中抽出；牌保持原位并加强光晕，确认后直接化为能量流进入对应牌位。

### 落牌

- 删除 `card-slot-mist.png` 的使用。
- 每个牌位由 CSS 多层效果构成：旋转能量环、径向光雾、粒子轨迹、牌形轮廓和确认脉冲。
- 未填牌位只显示克制的轮廓；当前牌位加强呼吸；已填牌位显示持续流动的能量核心。
- 落牌阶段不揭示牌面和牌意，只记录卡牌、正逆位与牌位。

### 统一揭示

- 最后一个牌位确认后进入 `REVEALING`，不渲染最后一张单独大牌。
- 所有已填牌位同时增强能量，三张或五张牌一起翻开。
- DeepSeek 请求与统一揭示动画并行启动，减少结果页等待时间。

## 结果页

### 三张牌阵

- 三张大牌以全宽三联布局占据主要画面，分别标明过去、现在、未来和正逆位。
- 每张牌显示简短本地牌意，不突出第三张。
- DeepSeek 综合解答使用问题和三张牌的完整上下文，放在横跨下方的玻璃阅读区。

### 五张牌阵

- 五张牌以大型五角星/星阵结构覆盖主舞台，不再缩在左侧小面板。
- 身、心、灵、情感、环境五个牌位与卡牌一一对应。
- 综合解答区与牌阵保持同屏；文字过长时只让解答区域内部滚动。

### 共同规则

- 整个结果页保持在 `100dvh` 内，不产生页面级上下滚动。
- 三张和五张都显示一段面向原问题的综合回答，而不是简单拼接单牌牌意。
- DeepSeek 尚未返回时显示“星域正在组织完整回应”。
- 调用失败或未配置 Key 时，仍展示所有牌和本地牌意，并提供“重新生成综合解答”按钮。

## DeepSeek 数据流

1. 页面在全部牌位确认后提交：问题、牌阵类型、卡牌 ID、正逆位和牌位 ID。
2. 新增服务端 Route Handler，校验牌阵数量、卡牌 ID、牌位与正逆位。
3. 服务端根据本地 78 张牌数据补齐牌名、关键词和牌意；客户端不直接提交可伪造的牌意文本。
4. 服务端使用 `DEEPSEEK_API_KEY` 和可配置的 `DEEPSEEK_MODEL` 调用 DeepSeek。密钥不进入 `NEXT_PUBLIC_*` 变量，也不返回浏览器。
5. 提示词要求使用中文、围绕用户问题、综合所有牌位、避免宿命化断言，并保留医疗、法律、财务等高风险领域的边界说明。
6. 接口向前端返回一段综合解读；前端维护 `idle / loading / success / error` 状态并支持重试。

实现前需要根据 DeepSeek 官方当前文档确认接口地址、模型名、请求格式和错误结构，不在设计阶段硬编码可能变化的外部契约。

## 原审查问题修复

### 点击模式牌环

- 缓存由 `results` 派生的 `drawnCardIds`，只有抽牌结果变化时才产生新数组。
- 避免 `ClickGestureAdapter` 动画 effect 被普通渲染反复清理，恢复连续牌环运动。

### 移动端摄像头控制

- 小于 768px 时保留紧凑的“改用点击模式 / 重试摄像头”按钮，不再隐藏。

### 无效整页渲染

- 只有当前确实存在错误时才派发 `CLEAR_ERROR`。
- reducer 对空错误清理直接返回原状态。
- MediaPipe 调试面板只在状态内容真正变化时更新。

### 每日一签异常存储

- `selectedCardId` 与 `orientation` 必须同时为空或同时有效。
- 不一致记录自动按当天新记录恢复。

## 组件与职责

- `GestureDrawPage`：业务状态、DeepSeek 请求状态和最终页面编排。
- `GestureController` / `GestureEngine` / `ClickGestureAdapter`：输入转换，不持有牌阵业务状态。
- `TarotUniverse`：洗牌、选牌、落牌和统一揭示视觉，不直接决定状态转换。
- `CardOrbit` / `TarotCard3D`：宽幅扇面、候选光晕和卡牌表现。
- `SpreadBoard`：三张与五张牌位能量效果。
- `ResultSpace`：按牌阵类型渲染全屏结果布局和 DeepSeek 综合解答。
- 服务端解读 Route Handler：验证请求、组织牌面上下文、调用 DeepSeek、规范化错误。

## 验证标准

1. 三张牌阵最后一张确认后，三张牌同时揭示，不出现第三张单独牌意画面。
2. 五张牌阵最后一张确认后，五张牌同时揭示，不出现第五张单独牌意画面。
3. 1280×720 下洗牌与选牌明显覆盖主舞台，扇面接近全宽。
4. 候选牌没有牵引线，也不会从牌群中单独抽出。
5. 已抽牌位不再请求或显示 `card-slot-mist.png`。
6. 三张结果页显示三张完整牌和综合解答；五张结果页显示大型五角星牌阵和综合解答。
7. DeepSeek 成功、加载、失败和重试四种状态均可验证；未配置 Key 时页面不崩溃。
8. 点击模式横向牌环连续运动，移动端模式按钮可见。
9. `npm run lint`、`npx tsc --noEmit`、`npm run build` 全部通过。

## 非目标

- 不调整 MediaPipe 手势分类阈值。
- 不引入 Three.js 或 R3F。
- 不接入登录、数据库或历史记录保存。
- 不新增项目级端到端测试依赖。
- 不处理与本次手势抽牌体验无关的占位页面。
